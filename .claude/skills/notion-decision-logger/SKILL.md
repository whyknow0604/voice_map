# Notion Decision Logger

Notion의 의사결정 로그 DB에 프로젝트의 중요 결정 사항을 기록한다.

## 언제 사용하는가

- 팀원이 L2 의사결정을 tech-lead에게 보고할 때
- tech-lead가 아키텍처, 스프린트 관련 결정을 내릴 때
- 스프린트 플래닝에서 기술적 방향을 결정할 때

## 대상 Notion DB

### 05. 의사결정 로그
- DB ID: `c0a23822df13405eb318b4c42079479a`

## 기록할 프로퍼티

| 프로퍼티 | 타입 | 설명 |
|---------|------|------|
| 결정 사항 | title | 결정의 한 줄 요약 |
| 결정 주체 | select | Tech Lead / Frontend / Backend / AI Knowledge |
| Level | select | L1-자동 / L2-즉시보고 |
| 카테고리 | select | 기술 / 기획 / 운영 / 실험 |
| 배경 | text | 왜 이 결정이 필요했는가 |
| 결정 내용 | text | 구체적으로 무엇을 선택했는가 |
| 트레이드오프 | text | 검토한 대안과 선택하지 않은 이유 |
| 결과 | select | 검증중 (기본값) / 유효 / 번복 |
| 날짜 | date | 결정 날짜 |
| 관련 태스크 | relation → 06. 서브태스크 | 어떤 태스크 수행 중 발생한 결정인지 |

## 실행 절차

### L2 보고 수신 시
1. 팀원의 message에서 다음을 추출한다:
   - 무엇을 결정했는지
   - 왜 그렇게 결정했는지
   - 어떤 대안을 검토했는지
2. 추출한 정보가 불충분하면 팀원에게 추가 정보를 요청한다
3. Notion MCP로 의사결정 로그에 새 항목을 생성한다
4. 관련 태스크가 있으면 relation을 연결한다

### tech-lead 직접 결정 시
1. 결정의 맥락과 내용을 정리한다
2. Notion MCP로 의사결정 로그에 기록한다
3. 영향받는 팀원에게 message로 결정 사항을 전달한다

## 기록 품질 기준

좋은 의사결정 로그는 나중에 "왜 이렇게 되어 있지?"라는 질문에 답할 수 있어야 한다.

- **배경**: 단순히 "필요해서"가 아니라, 어떤 상황에서 어떤 문제가 있었는지
- **트레이드오프**: 최소 하나의 대안과 그 대안을 선택하지 않은 이유
- **결정 내용**: 구체적이고 실행 가능한 수준으로 (모호한 방향성이 아닌 명확한 선택)
