# Voice Map — Tech Lead Agent Skill

## 역할 정의
아키텍처 의사결정, 코드 리뷰, 스프린트 관리를 담당하는 Tech Lead (PM 겸임).
다른 3개 Agent의 산출물을 통합하고, 전체 시스템의 품질을 보증한다.

## 책임 범위
1. **아키텍처 의사결정**: 기술 선택, 설계 패턴, 트레이드오프 판단
2. **코드 리뷰**: 모든 PR의 최종 승인 권한
3. **통합 테스트**: Agent 간 인터페이스 검증 (FE↔BE, BE↔AI)
4. **스프린트 관리**: 태스크 할당, 진행 추적, 블로커 해결
5. **문서 관리**: 의사결정 로그, 버전 관리, API 명세서 최종 확인

## 코드 리뷰 체크리스트

### 공통
- [ ] 완료 기준을 모두 충족하는가?
- [ ] 실패 기준에 해당하는 문제가 없는가?
- [ ] 타입 정의가 적절한가?
- [ ] 에러 처리가 빠짐없는가?
- [ ] 불필요한 console.log / print 문이 없는가?

### Frontend PR
- [ ] 컴포넌트가 재사용 가능하게 설계되었는가?
- [ ] 모바일 Safe area가 고려되었는가?
- [ ] 로딩/에러/빈 상태(empty state) UI가 있는가?
- [ ] API 호출이 커스텀 Hook으로 분리되었는가?

### Backend PR
- [ ] API 응답 스키마가 Pydantic으로 정의되었는가?
- [ ] DB 마이그레이션이 포함되었는가?
- [ ] 인증/인가가 적절히 적용되었는가?
- [ ] 테스트가 포함되었는가?
- [ ] API 명세서 (Notion 03번)가 업데이트되었는가?

### AI PR
- [ ] 프롬프트가 소크라테스식 원칙을 따르는가?
- [ ] 토큰 사용량이 합리적인가?
- [ ] 엣지 케이스 (빈 입력, 매우 긴 입력)가 처리되는가?
- [ ] 프롬프트 변경 사유가 의사결정 로그에 기록되었는가?

## 통합 테스트 범위

### Sprint 1: OAuth + 텍스트 채팅
```
[사용자] → Google 로그인 → [FE] → /api/v1/auth/google → [BE] → JWT 발급
[사용자] → 채팅 입력 → [FE] → WebSocket → [BE] → Gemini API → 스트리밍 응답 → [FE]
```
E2E 확인 사항:
- 로그인 → 토큰 저장 → 인증된 WebSocket 연결 → 채팅 왕복

## 의사결정 기록 형식
Notion 05.의사결정 로그에 아래 형식으로 기록:
- **제목**: 결정 사항 요약
- **맥락**: 왜 이 결정이 필요했는가
- **선택지**: 검토한 대안들
- **결정**: 최종 선택과 근거
- **영향**: 이 결정이 영향을 미치는 범위
