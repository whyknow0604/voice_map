# Notion Spec Reader

Notion에서 프로젝트 스펙 문서를 읽어와 필요한 정보를 추출한다.

## 언제 사용하는가

- 팀원 spawn 시: 해당 팀원에게 필요한 스펙을 읽어서 spawn prompt에 포함
- 팀원의 스펙 참조 요청 시: 특정 기능이나 API의 최신 명세를 조회
- 스프린트 플래닝 시: 구현할 기능의 상세 스펙 확인

## 참조하는 Notion 문서

### 01. 서비스 기획서 (페이지)
- Page ID: `32cb1ba6f41881888710ef66f3858426`
- 내용: 한 줄 정의, 타깃, 문제 정의, 핵심 가치 제안, 기능 구조, 성공 지표

### 02. 기능명세서 (DB)
- DB ID: `fae77560a88140b3b54652a2e208fabf`
- 프로퍼티: 기능명, 기능 설명, 담당 Agent, 상태, 우선순위, 사용자 입력, 출력 데이터, 화면

### 03. API 명세서 (DB)
- DB ID: `4deb7187a03e419f8a60710301bf7349`
- 프로퍼티: 엔드포인트, 메서드, 상태, 설명, 요청 파라미터, 응답 데이터, 관련 기능

## 에이전트별 필요 스펙

팀원을 spawn할 때 어떤 스펙을 전달해야 하는지:

### frontend에게
- 서비스 기획서 중 기능 구조 (MVP 범위)
- 기능명세서에서 배정된 기능의 화면 설명, 사용자 입력/출력
- API 명세서에서 관련 엔드포인트의 요청/응답 구조

### backend에게
- 기능명세서에서 배정된 기능의 상세 설명
- API 명세서에서 배정된 엔드포인트 전체
- ai-knowledge와의 인터페이스 정의 (해당 기능이 있을 경우)

### ai-knowledge에게
- 서비스 기획서 중 핵심 가치 제안, 성공 지표
- 기능명세서에서 AI 관련 기능의 상세 설명
- 실험 레이어 (Graph RAG) 관련 스펙

### qa에게
- 서비스 기획서 전체 (사용자 관점 이해)
- 이번 스프린트에서 구현된 기능 목록과 완료 기준
- API 명세서 (API 통합 테스트용)

## 실행 절차

1. 어떤 팀원에게 어떤 스펙이 필요한지 판단한다
2. Notion MCP로 해당 문서/DB를 조회한다
3. 팀원에게 필요한 부분만 추출하여 정리한다
4. spawn prompt에 포함하거나, message로 전달한다
